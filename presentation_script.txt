Presenter 1: Introduction, Problem Statement, and Project Overview

"Good morning, respected examiner. We are presenting the SRIMCA AI Assistant.

Our project solves existing issues in Shrimad Rajchandra Institute of Management and Computer Application.
Currently, most student and visitor queries are handled manually by office staff and faculty.
This creates many problems:
- students cannot get information 24×7,
- office staff have too much manual work,
- important notices are not always delivered on time,
- there is no AI-based system to answer questions instantly,
- visitor registration is done manually and takes time,
- communication between students, faculty, and administration is slow.

To solve this, we developed the SRIMCA AI Assistant, a mobile app that offers:
- instant AI-based answers,
- 24×7 access to college information,
- real-time notifications for notices, events, and approvals,
- secure login with role-based access,
- QR-based visitor registration and visitor management.

This project is built with:
- Flutter for the frontend mobile app,
- Python Flask for the backend REST API,
- MongoDB Atlas for the database,
- JWT for authentication,
- Firebase services for push notifications and optional auth integration.

The purpose is to provide instant, secure, and reliable access to information for students, faculty, and visitors, while reducing manual workload and improving communication.

The main project goal is to create a smart SRIMCA AI system that helps users find answers faster, manage visitors smoothly, and deliver notices effectively."

Presenter 3: Data Pipeline, Backend Architecture, and Technology Details

"Now I will describe the technical architecture, data pipeline, and all small backend details.

Data preprocessing starts from `backend/srimca/data/sample_srimca_info.txt`, where official SRIMCA content is collected.
The loader scripts in `backend/srimca/loader.py` use Python `requests` and `BeautifulSoup` to fetch website pages and parse content.
Then the backend loads and cleans the text data from `.txt` files and combines them into a single knowledge set.

Text chunking is handled in `backend/srimca/build_db.py`.
We split content by newlines and create knowledge chunks. Each chunk becomes a separate searchable item.
This results in about 30 or more text segments stored in MongoDB.

Embedding generation is configured in `backend/srimca/config.py`.
The model used is `sentence-transformers/all-MiniLM-L6-v2`, which creates 384-dimensional vectors locally.
This approach has no extra API cost and uses the open-source sentence-transformers library.

Vector storage and retrieval:
- We store each chunk with `text`, `embedding`, and metadata in the `knowledge` MongoDB collection.
- Vector search is enabled in MongoDB to perform similarity calculations.
- `backend/srimca/retriever.py` performs semantic search using cosine similarity.
- The top-k relevant chunks are retrieved as context for answer generation.

Answer generation happens in `backend/srimca/gpt_answer.py`.
It can use the OpenAI GPT API when available, otherwise it uses template-based fallback responses.
The response pipeline is:
- user submits question,
- backend receives API request,
- question is embedded,
- vector search returns relevant knowledge chunks,
- GPT or fallback generates the final answer,
- backend sends the response back to the app.

Backend technical stack and files:
- `backend/app.py` initializes Flask, CORS, and route blueprints.
- `backend/auth.py` handles registration, OTP, login, JWT generation, and email sending.
- `backend/database.py` initializes MongoDB connections and collections.
- `backend/config.py` loads `.env` variables and config values.
- Route files include `routes/notices.py`, `routes/assignments.py`, `routes/materials.py`, `routes/faqs.py`, `routes/users.py`, `routes/admin.py`, `routes/ai.py`, and `routes/notifications.py`.

Important API endpoints:
- `/api/send-registration-otp`
- `/api/verify-registration-otp`
- `/api/register`
- `/api/login`
- `/api/notices`
- `/api/ai/chat`
- `/api/users`, `/api/admin/users`
- notification and content management endpoints.

Frontend technical details:
- `lib/api_service.dart` manages all REST calls, including OTP, register, login, notices, AI queries, and user data.
- `lib/registration_otp_page.dart` handles OTP sending, verification, resend cooldown, and registration completion.
- Important UI files: `login_register_screen.dart`, `chat_screen.dart`, `faculty_dashboard.dart`, `admin_main_dashboard.dart`, `visitor_inquiry_page.dart`, `faculty_upload_notice_page.dart`, and `faculty_notifications_page.dart`.
- Flutter packages used:
  - `firebase_core`, `firebase_auth`, `cloud_firestore`, `firebase_messaging`, `flutter_local_notifications`,
  - `http`, `shared_preferences`, `app_links`, `fl_chart`, `intl`, `file_picker`.

Development tooling and environment:
- Flutter SDK and Dart 3.10.8
- Python 3 backend with Flask 3.0.0
- MongoDB Atlas database
- Git and GitHub for version control
- Postman for API testing
- Android Studio and VS Code for development
- Deployment via Render and Firebase Hosting.

Small but important files:
- `backend/requirements.txt` lists backend dependencies like Flask, pymongo, bcrypt, PyJWT, firebase-admin, sentence-transformers, and OpenAI.
- `backend/.env.example` defines environment variables for MongoDB, JWT, Flask, Firebase, and SMTP.
- `pubspec.yaml` defines Flutter dependencies and assets.
- `README.md`, `TODO.md`, and `table.txt` are documentation and project reference files.

This architecture combines AI, mobile, backend, and database technologies to deliver a complete SRIMCA assistant experience."
